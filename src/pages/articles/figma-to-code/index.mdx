## üß± Step 1: Extract and Normalize Design Tokens

Design tokens are the foundation for scalable design systems. They represent visual values such as:

- Colors (`primary`, `neutral-100`)
- Typography (`font-size-sm`, `line-height-lg`)
- Spacing (`spacing-4`, `padding-md`)
- Elevation, radius, z-index, opacity

### üîß How to extract from Figma

Use plugins like:

- [Figma Tokens](https://www.figma.com/community/plugin/843461159747178978)
- [Design Token Exporter](https://www.figma.com/community/plugin/876619244393011362)

These tools let you export token values as JSON.

Example Figma token JSON:

```json
{
  "color": {
    "primary": "#3364dd",
    "background": "#ffffff"
  },
  "spacing": {
    "md": "1rem"
  },
  "typography": {
    "body": {
      "fontSize": "16px",
      "lineHeight": "24px"
    }
  }
}
```

## üí° Step 2: Map Tokens to CSS Custom Properties

Once exported, you can translate those tokens into `:root` CSS variables or preprocess them using tools like [Style Dictionary](https://amzn.github.io/style-dictionary/).

```css
:root {
  --color-primary: #3364dd;
  --spacing-md: 1rem;
  --font-body-size: 1rem;
  --font-body-line-height: 1.5rem;
}
```

Use these variables throughout your SCSS or utility classes for consistency.

## üß© Step 3: Structure Your Component Library

Use atomic or component-driven thinking:

- `Atoms`: buttons, inputs, badges
- `Molecules`: input groups, cards
- `Organisms`: forms, navbars, modals

For each component:

1. Reference your Figma components.
2. Match padding, border, size, and typography.
3. Code reusably ‚Äî avoid one-off classes unless explicitly needed.

Example SCSS for a `Button`:

```scss
.btn {
  padding: var(--spacing-md);
  font-size: var(--font-body-size);
  border-radius: 0.5rem;
  background-color: var(--color-primary);
  color: white;
  border: none;
  cursor: pointer;
  transition: background-color 0.2s ease;

  &:hover {
    background-color: #2850ba;
  }
}
```

## üìê Step 4: Match Layouts and Grids

Check your Figma grid setup:

- How many columns?
- What‚Äôs the gutter size?
- Are there breakpoints defined?

Then replicate with a utility framework or custom layout system.

Tailwind example:

```html
<div class="grid grid-cols-12 gap-4">
  <div class="col-span-6">Left Column</div>
  <div class="col-span-6">Right Column</div>
</div>
```

Custom CSS Grid:

```css
.container {
  display: grid;
  grid-template-columns: repeat(12, 1fr);
  gap: 1rem;
}
```

## üö´ What to Avoid

- Overwriting tokens in components (use the variables!)
- Implementing visual tweaks that don't match the system
- Ignoring responsive specs from the design
- Using hardcoded values without documentation
- Building without accessibility in mind (ARIA, focus, contrast)
